# Backend «Карты скорби»

REST-backend для раздела «Карта скорби» энциклопедии Бензина: список заправок
по всей России (реальные данные из OpenStreetMap) и общие для всех
посетителей сайта отметки статуса ("есть/нет топлива", очередь, закрыта).

## Почему Go, а не Java/Spring Boot и не PHP

Изначально рассматривался Spring Boot, но выбор был пересмотрен по факту
реальных условий сервера `37.140.192.133` (shared-хостинг reg.ru,
ISPmanager, AlmaLinux 8.10, пользователь `u3577787` без root/sudo):

- Docker установлен (`docker --version` → 1.13.1), но **демон не запущен**,
  а поднять его не получится без root (`sudo` требует пароль, которого нет).
  Значит, Docker-путь на этом конкретном сервере в принципе недоступен —
  ни Java, ни Go, ни любой другой стек не запустится в контейнере здесь.
- На сервере **нет Java/JRE и нет Maven** (`java`, `mvn` → command not found).
  Для Java пришлось бы либо тащить на сервер JDK (~100+ МБ, user-space без
  root, отдельная головная боль с обновлениями), либо собирать через Docker
  (см. пункт выше — недоступно).
- Оперативной памяти на сервере на самом деле много (125 ГБ, ~89 ГБ
  доступно) — то есть JVM-overhead сам по себе не был бы фатальным
  ограничением, но простота деплоя перевешивает: Go даёт **один статический
  бинарник linux/amd64**, собранный кросс-компиляцией на любой машине
  разработчика (`GOOS=linux GOARCH=amd64 go build`), без каких-либо
  зависимостей на сервере — ни JVM, ни интерпретатора, ни virtualenv.
  Запуск — это `scp` + `nohup ./benzinopedia-backend &`.
- Python тоже был вариантом (на сервере есть `python3`, хоть и старый
  3.6.8), но потребовал бы либо системного питона с ограниченным набором
  пакетов, либо virtualenv/pip с зависимостями (FastAPI/Flask, SQLAlchemy,
  Alembic, драйвер MySQL) — то есть больше поверхности для сломаться на
  чужом сервере, чем один бинарник.
- PHP как язык **бизнес-логики backend'а не использовался** — это было
  прямым требованием задачи. PHP использован только как тонкий
  инфраструктурный мостик, см. раздел ниже.

Итог: **Go** — весь REST API, вся работа с БД, все автомиграции, вся
валидация — находится в Go-коде в этой директории (`main.go`,
`internal/...`). Никакой бизнес-логики в PHP нет.

## Зачем всё-таки существует `deploy/api.php`

Это НЕ backend и не часть API-логики — это прозрачный HTTP-прокси
("шim"), решающий чисто инфраструктурную проблему конкретно этого сервера:

1. Файрвол сервера снаружи пропускает только стандартные веб/почтовые порты
   (80, 443, 22, 25 и т.п.) — практически проверено: тестовый HTTP-сервер,
   поднятый на порту 8082, был недоступен снаружи (`curl` до
   `37.140.192.133:8082` таймаутился), хотя тот же порт открыт и слушает
   локально на сервере. Открыть порт в файрволе для проекта нельзя: нет
   root/sudo, а `firewall-cmd`/`iptables` тоже не установлены/не доступны
   пользователю.
2. Единственный публичный вход на домен `benzinopedia.ru` — это nginx,
   конфиг которого (`/etc/nginx/vhosts/u3577787/benzinopedia.ru.conf`)
   принадлежит `root:root` (права `644`, есть только чтение) и содержит
   **фиксированный** `proxy_pass http://127.0.0.1:8080;` — это встроенный
   в ISPmanager проксирующий Apache/PHP-FPM для этого аккаунта, и добавить
   туда свой `location /api { proxy_pass http://127.0.0.1:8082; }`
   невозможно без root-доступа к nginx-конфигурации.
3. При этом `.php`-файлы в веб-корне (`/var/www/u3577787/data/www/benzinopedia.ru/`)
   обслуживаются штатно через этот же nginx → Apache/PHP-FPM механизм — то
   есть единственный доступный пользователю без root способ добавить свою
   логику "перед" публичным HTTPS сервера — это положить `.php`-файл в
   веб-корень.

Поэтому `deploy/api.php` — это ровно 100 строк, которые: принимают HTTP-
запрос на `https://benzinopedia.ru/api.php/...`, прозрачно перекладывают
его на `http://127.0.0.1:8082/api/...` (где слушает настоящий Go-backend,
привязанный только к loopback-адресу, недоступному снаружи напрямую — что
даже безопаснее прямого проброса порта), и возвращают ответ как есть.
Никакой работы с БД, валидации статусов, бизнес-правил в этом файле нет —
всё это в Go. Заодно этот же файл решает CORS (запрос идёт на тот же
домен, что и сам сайт) и mixed content (путь `/api.php` доступен по
тому же https, что и сам сайт, независимо от того, что backend внутри
сервера общается по обычному http).

**Если у вас VPS с root-доступом** — этот `api.php`-шim не нужен вообще:
достаточно добавить в конфиг nginx/Apache обычный `proxy_pass` на порт
backend'а (см. `docker-compose.yml`/`Dockerfile` ниже) и настроить свой
файрвол на проброс нужного порта — тогда фронтенд можно завести прямо на
`https://ваш-домен/api/...` без какого-либо PHP в цепочке.

## Автомиграции БД

Схема БД версионируется SQL-файлами в `migrations/*.up.sql` /
`*.down.sql`, вшитыми в бинарник через `go:embed` (см. `main.go`) и
применяемыми на каждом старте приложения через
[`golang-migrate/migrate`](https://github.com/golang-migrate/migrate) —
это прямой аналог Flyway для Go: миграции трекаются в таблице
`schema_migrations`, при следующем перезапуске уже применённые миграции
не повторяются.

- `0001_init.up.sql` — таблицы `stations` и `status_records`.
- `0002_seed_demo_stations.*` — **не используется в проде** (черновой
  вариант с 5-7 демо-заправками, оставлен для истории/тестов).

Реальное заполнение станций — не через SQL-миграцию, а через отдельный шаг
сидирования (`internal/seed/seed.go`), который проверяет `SELECT COUNT(*)
FROM stations` и, если таблица пуста, заливает станции из встроенного
JSON-снэпшота `data/stations_ru.json` (тоже через `go:embed`, тоже
срабатывает автоматически при старте, но ровно один раз).

## Откуда взялись реальные заправки

`data/stations_ru.json` собран через **Overpass API** (публичный,
бесплатный API поверх данных OpenStreetMap, без ключа) запросами по всем
~85 регионам России (`admin_level=4`) — то есть это не только областные
центры, а вся страна целиком, все населённые пункты, где в OSM отмечены
заправки (`amenity=fuel`). Точные цифры (сколько регионов, сколько
станций, сколько заняло по времени) — в финальном отчёте задачи и в
`data/import_report.json` рядом со снэпшотом.

Данные распространяются по лицензии **ODbL**, атрибуция
"© OpenStreetMap contributors" обязательна и выведена прямо на странице
сайта под картой.

У всех импортированных станций **сознательно нет ни одной начальной
отметки статуса** — открытого источника данных о реальном наличии
топлива не существует, поэтому статус остаётся "неизвестен" (серый
маркер), пока его не оставит реальный посетитель сайта через
`POST /api/stations/{id}/statuses`. Ради этого весь backend и делался.

## REST API

| Метод | Путь | Описание |
|---|---|---|
| GET | `/api/health` | health-check |
| GET | `/api/stations` | облегчённый список всех заправок (id, название, координаты, статус последней отметки) — специально без адреса/комментариев, чтобы ответ оставался быстрым при десятках тысяч записей |
| GET | `/api/stations/{id}` | полная карточка одной заправки: адрес, последняя отметка целиком, до 5 последних отметок именно по этой заправке |
| POST | `/api/stations` | создать новую заправку (клик по свободному месту на карте), опционально сразу с первой отметкой статуса |
| POST | `/api/stations/{id}/statuses` | добавить отметку статуса существующей заправке — основной эндпоинт крауд-сорсинга |
| GET | `/api/statuses/latest?limit=5` | последние N отметок по всем заправкам сразу — для блока "Последние отметки" на сайте |

Enum статусов: `AVAILABLE`, `SHORTAGE_92`, `SHORTAGE_95`, `QUEUE_ONLY`,
`CLOSED`.

## Переменные окружения

| Переменная | Назначение | Дефолт |
|---|---|---|
| `MYSQL_HOST` | хост MySQL | `127.0.0.1` |
| `MYSQL_PORT` | порт MySQL | `3306` |
| `MYSQL_DATABASE` | имя базы | `u3577787_default` |
| `MYSQL_USER` | пользователь БД | `u3577787_default` |
| `MYSQL_PASSWORD` | пароль БД | (реальный пароль сервера, см. задачу) |
| `SERVER_PORT` | порт, на котором слушает Go-backend | `8082` |
| `CORS_ALLOWED_ORIGINS` | список origin'ов через запятую, `*` — разрешить все | `https://benzinopedia.ru,https://oleg-rakitin.github.io,...` |

## Сборка и локальный запуск

```bash
go build -o benzinopedia-backend .
MYSQL_PASSWORD=... ./benzinopedia-backend
```

Кросс-компиляция для сервера (Linux amd64) с любой машины разработчика:

```bash
GOOS=linux GOARCH=amd64 go build -o benzinopedia-backend-linux .
```

## Docker (готов, но НЕ используется в проде на текущем сервере)

`Dockerfile` и `docker-compose.yml` полностью рабочие — многостадийная
сборка (golang:1.24 → distroless static), `network_mode: host` в compose
(чтобы `127.0.0.1` внутри контейнера был тем же `127.0.0.1`, на котором
слушает MySQL хоста — без этого режима `localhost` внутри контейнера был
бы отдельным сетевым namespace контейнера, а не хоста, и подключение к
БД просто не работало бы). Образ собирается и проверен локально
(`docker build .` — успешно). На сервере `37.140.192.133` не
запускался и не может быть запущен, потому что там нет работающего
Docker-демона и нет root, чтобы его поднять (см. раздел про Go выше).

Готов к использованию "как есть" на любом VPS с root-доступом и
установленным/запущенным Docker:

```bash
docker compose up -d --build
```

## Деплой на текущий shared-хостинг (реально использованный способ)

1. Кросс-компиляция бинарника на машине разработчика.
2. `scp` бинарника на сервер в домашнюю директорию пользователя.
3. Запуск через `nohup ./benzinopedia-backend > backend.log 2>&1 &`
   (или `screen`), слушает `127.0.0.1:8082` — то есть недоступен снаружи
   напрямую, что нормально: наружу его выставляет `deploy/api.php` (см.
   выше), а не файрвол.
4. `deploy/api.php` копируется в веб-корень `benzinopedia.ru` как `api.php`.
5. Автозапуск при перезагрузке сервера через `crontab -e` с `@reboot`
   (или systemd `--user`-юнит, если на сервере включен linger для
   пользовательских сервисов) — см. `deploy/benzinopedia-backend.cron`.
